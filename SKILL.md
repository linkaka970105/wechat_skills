# WeChat Shop Data

使用微信小店官方开放接口完成后台数据查询与分析。默认优先只读查询，避免模型自行决定大规模明细补拉策略。

## 工作原则

1. 先判断需求属于哪个领域：订单、商品、库存、赠品、营销活动。
2. 先判断任务模式：摘要查询、限定明细、全量导出。
3. 优先使用列表接口或批量接口，避免不受控的逐条详情调用。
4. 只有在用户明确需要完整字段时，才进入明细补全。
5. 涉及写操作时，必须先说明影响范围并等待确认。
6. 返回结果时优先输出结构化摘要，而不是原始接口全文。

## 任务模式

### `summary_only`

适用于：

- 看最近订单状态
- 统计售后单数量
- 查看商品列表
- 查看活动列表

处理规则：

- 只调用列表或搜索接口
- 返回数量、ID、状态、分页信息
- 不默认全量补详情

### `detail_on_demand`

适用于：

- 查看命中结果里的完整字段
- 查询小批量订单/商品/赠品详情

处理规则：

- 先通过列表接口收集 ID
- 仅对小集合补详情
- 明确告知当前是“限定补全”，不是全量补全

### `full_export`

适用于：

- 一个月订单完整导出
- 对账、审计、归档
- 大范围完整明细拉取

处理规则：

- 先分页收集全部 ID
- 去重后分块处理详情
- 使用固定并发、限流、重试、续跑策略
- 对外说明当前是批处理导出模式

## 固定扩展阈值

为了避免不同模型的效率差异过大，详情补全遵循固定规则：

- 命中数量 `<= 20`：允许自动补详情
- 命中数量 `21-200`：先返回摘要，再说明可以继续补详情
- 命中数量 `> 200`：默认进入批处理/导出模式，不在一次普通对话里同步逐条补完

不要让模型自行临场决定是否补 100、500、1000 条详情。

## 统一数据获取策略

### 订单

- 列表阶段：`getorderlist`、`searchorder`
- 明细阶段：`getorder`
- 规则：
  - 列表接口主要用于收集 `order_id`
  - 若用户要完整字段，进入明细补全
  - 月度订单导出必须按 7 天时间窗切片，因为列表接口单次时间范围不能超过 7 天

### 商品

- 列表阶段：`getproductlist`
- 明细阶段：`getproduct`
- 规则：
  - 列表接口返回 `product_id`
  - 需要标题、类目、SKU、属性等完整字段时，再补详情

### 赠品

- 列表阶段：`getgiftproductlist`
- 明细阶段：`getgiftproduct`
- 规则：
  - 列表接口返回赠品 ID
  - 需要完整赠品信息时，再补详情

### 库存

- 单条查询：`getstock`
- 批量查询：`batchgetstock`
- 流水查询：`getstockflow`
- 规则：
  - 多商品、多 SKU 库存查询默认优先 `batchgetstock`
  - 不要把库存场景误写成“对每个 SKU 调一次 `getstock`”的低效模式

### 营销活动

- 赠品列表：`getgiftproductlist`
- 限时抢购任务列表：`getlimiteddiscounttasklist`
- 规则：
  - 如果列表结果已满足问题，就直接返回
  - 只有用户明确需要单条完整信息时，才补单条详情

## 大任务批处理规范

当进入 `full_export` 或大批量 `detail_on_demand` 时，遵循固定流程：

1. 先按分页接口拉取 ID 列表。
2. 保留并传递 `next_key`。
3. 合并并去重所有 ID。
4. 按固定大小分块处理。
5. 以固定并发拉取详情。
6. 遇到限流或临时失败时退避重试。
7. 返回当前进度、已完成数量、剩余数量。

推荐执行参数：

- 分块大小：20 到 50
- 并发数：5 到 10
- 限流后策略：降低并发并退避重试

这里的重点是“固定策略”，而不是让模型自由发挥。

## 月度订单导出推荐流程

如果用户要一个月订单的完整字段，按以下流程处理：

1. 把一个月拆成多个 7 天时间窗。
2. 对每个时间窗调用 `getorderlist`。
3. 用 `next_key` 翻完每个时间窗的全部分页。
4. 汇总并去重全部 `order_id`。
5. 分块并发调用 `getorder`。
6. 统一整理为导出结构。

典型导出字段：

- 商品名称
- 单价
- 数量
- 售后信息
- 实收款
- 优惠信息
- 买家信息
- 收货信息
- 订单状态

## 输出格式建议

优先按以下结构输出：

- 查询目标
- 任务模式
- 使用接口
- 筛选条件
- 结果摘要
- 是否执行详情补全
- 分页信息
- 风险提示
- 下一步建议

## 写操作安全规则

以下能力属于写操作，默认不直接执行：

- 修改订单价格
- 修改订单备注
- 修改订单地址
- 修改物流信息
- 更新库存
- 商品新增、更新、删除、上下架
- 创建、停止、删除赠品活动
- 创建、停止、删除限时抢购任务

执行前必须：

1. 明确说明这是写操作。
2. 说明影响哪些订单、商品、库存或活动。
3. 获得明确确认后再执行。

## 注意事项

- 不要把所有列表接口都默认理解成“结果已完整”。
- 也不要把所有列表接口都默认理解成“必须立刻逐条补详情”。
- 先按任务模式选策略，再决定是否补详情。
- 对高频运营查询，优先摘要返回；对导出任务，再进入批处理模式。
- 分页查询时必须保留 `next_key`，避免漏数或重复。
